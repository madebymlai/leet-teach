The Hash-Map Lookup Pattern

A family of problems, one idea: instead of re-scanning the data to find a match, remember what you have already seen in a hash map so a lookup takes O(1) instead of O(n). Trade memory for speed.

Spot the family

A problem belongs to this family when you can phrase it as:

“For each element x, is there some related element (a complement, a duplicate, a previous occurrence, a count) that I could answer in one step if I had remembered what came before?”

Signal words in the statement: pair, duplicate, first recurring, count of, anagram of, two … that …. If you can name a complement - the thing x needs in order to satisfy the condition - you are in this family.

Notice the shape: one pass, one dict, look up the complement, then record the current element. Only the complement and the stored value change between problems.

The shape, abstracted

seen = {}                       # the memory
for item in stream:
    complement = needed_relation(item)
    if complement in seen:      # O(1) - the move that beats the nested loop
        return answer_from(seen[complement], item)
    seen[item] = something      # record AFTER checking
Why record after, not before?

Two reasons. (1) Self-match: in Two Sum you may not use the same element twice, so x must not find itself. (2) Semantics: seen means "elements before this one." Recording first makes the current element look like a prior one.

Choosing the key and the value

The dict answers one question in O(1): "is this key present?" So the key is the thing you are asking about - the complement you look up. The value is whatever you need to retrieve when the answer is "yes."

Rule

The in check tests the keys. So ask: what does complement in seen need to test? That thing is your key. Whatever you need back from seen[complement] to build the answer is your value.

Walk it from the in check, not from the data:

  • Two Sum - "is target - x a number I have seen?" Key: the number. Value: its index (you must return indices).
  • Contains Duplicate - "is x a value I have seen?" Key: the value. Value: anything (a bool, a count) - the problem only asks yes/no.
  • First Unique Character - "how many times have I seen c?" Key: the character. Value: its count.
If you key by the index, then value in seen asks "is this value an index I have visited?" - a different question than the one the problem poses. The test cases may pass by coincidence; the logic is wrong.

What the pattern costs

Time drops from O(n²) (nested loop, re-scan each step) to O(n): one pass, each dict operation averaging O(1).

Space rises from O(1) to O(n): in the worst case you store every element before finding the answer.

This trade - pay memory, buy speed - is one of the most common moves in algorithm problems. When you see a brute force that re-scans, ask: what would I need to remember to make each step O(1)?

Worked example: Two Sum

For each x at index i, the complement is target − x. The stored value is the index, because the problem asks for indices.

def twoSum(self, nums, target):
    seen = {}                       # number -> index
    for i, x in enumerate(nums):
        complement = target - x
        if complement in seen:
            return [seen[complement], i]
        seen[x] = i

The nested loop checked every (i, j) pair: O(n²). The dict remembers each number so a later element finds its partner in one lookup.

For Contains Duplicate (“does any value appear twice?”), what is the complement and what do you store?
  • Complement: a different index with the same value. Store: index -> value.
  • Complement: the value itself (already seen?). Store: value -> True.
  • Complement: target − value. Store: value -> count.
  • Complement: the next element. Store: value -> next value.
What is the complexity trade this pattern makes, relative to the nested-loop brute force?
  • Time O(n²) -> O(n); space unchanged O(1).
  • Time unchanged; space O(n) -> O(1).
  • Time O(n²) -> O(n); space O(1) -> O(n).
  • Both time and space improve.
